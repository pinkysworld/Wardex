# Supply-Chain Attestation Inputs

Design document for R20 (Verifiable Supply-Chain Attestation for Firmware and Models).

## Goals

1. Prove that the running Wardex binary matches a known-good vendor-signed build.
2. Verify that detection model artifacts (baselines, Wasm plugins) have not been tampered with since their signing.
3. Define the attestation data flow from build system through deployment to runtime verification.
4. Operate without mandatory network connectivity вҖ” local verification must be self-contained.

## Threat model

| Threat | Scenario |
|--------|----------|
| Binary replacement | Attacker replaces the `wardex` binary with a modified version |
| Model poisoning at rest | Adversary modifies persisted baseline files between runs |
| Plugin injection | Unauthorized Wasm plugin placed in the plugins directory |
| Build pipeline compromise | CI/CD produces a signed-but-malicious binary |
| Downgrade attack | Attacker reverts to an older, vulnerable binary version |

## Data structures

### Build manifest

Generated by the CI pipeline for every release build.

```rust
pub struct BuildManifest {
    /// Semantic version of Wardex (e.g., "0.2.0").
    pub version: String,
    /// Git commit hash of the source tree.
    pub commit: [u8; 20],
    /// UTC timestamp of the build.
    pub build_time: u64,
    /// Target triple (e.g., "aarch64-unknown-linux-gnu").
    pub target: String,
    /// SHA-256 hash of the final binary.
    pub binary_hash: [u8; 32],
    /// Merkle root of all source files included in the build.
    pub source_merkle_root: [u8; 32],
    /// SHA-256 hashes of bundled model/config artifacts.
    pub artifact_hashes: Vec<ArtifactEntry>,
    /// Ed25519 signature of the above fields by the build key.
    pub signature: [u8; 64],
    /// Public key of the signer (for key rotation verification).
    pub signer_pubkey: [u8; 32],
}

pub struct ArtifactEntry {
    /// Relative path of the artifact (e.g., "baseline.json").
    pub path: String,
    /// SHA-256 hash of the artifact content.
    pub hash: [u8; 32],
    /// Minimum acceptable version (for downgrade protection).
    pub min_version: u32,
}
```

### Local trust store

Resides on the device. Contains the set of trusted public keys and policy about what to accept.

```rust
pub struct TrustStore {
    /// Accepted signer public keys (Ed25519).
    pub trusted_keys: Vec<TrustedKey>,
    /// Minimum acceptable binary version (monotonically increasing).
    pub min_binary_version: u32,
    /// Whether to require attestation before the runtime processes any telemetry.
    pub require_attestation_at_boot: bool,
}

pub struct TrustedKey {
    pub pubkey: [u8; 32],
    /// Human-readable label (e.g., "release-1.0").
    pub label: String,
    /// Key valid-from timestamp.
    pub valid_from: u64,
    /// Key valid-until timestamp (0 = no expiry).
    pub valid_until: u64,
}
```

## Verification flow

### At boot

1. Read `/etc/wardex/manifest.json` (the build manifest shipped alongside the binary).
2. Verify the manifest signature against the local trust store's accepted keys.
3. Compute SHA-256 of the running binary (`/proc/self/exe` on Linux) and compare against `manifest.binary_hash`.
4. For each entry in `artifact_hashes`, compute SHA-256 of the on-disk file and compare.
5. Check `min_version` constraints against the trust store's `min_binary_version` (downgrade protection).
6. If any check fails:
   - Log a critical attestation failure to the audit log.
   - If `require_attestation_at_boot` is true, refuse to process telemetry.
   - If false, continue with a degraded trust level (logged but not blocking).

### At runtime (periodic)

1. Re-verify artifact hashes every N minutes (configurable).
2. If a baseline file changes (legitimate update via proof-carrying update), verify the new hash is bound to a valid proof.
3. If a Wasm plugin file changes unexpectedly, quarantine the plugin and alert.

## CI pipeline integration

The build pipeline produces the manifest as a build step:

```yaml
# .github/workflows/release.yml (excerpt)
- name: Build release binary
  run: cargo build --release --target ${{ matrix.target }}

- name: Generate attestation manifest
  run: |
    wardex-attest \
      --binary target/${{ matrix.target }}/release/wardex \
      --source-root . \
      --artifacts baseline.json,config.toml \
      --sign-key ${{ secrets.RELEASE_SIGNING_KEY }} \
      --output manifest.json

- name: Upload manifest
  uses: actions/upload-artifact@v4
  with:
    name: manifest-${{ matrix.target }}
    path: manifest.json
```

## Key rotation

1. New signing keys are distributed via signed key-rotation messages.
2. A rotation message is signed by the *old* key, contains the *new* public key, and specifies a validity window.
3. Devices accept the rotation only if the old key is still in the trust store and not expired.
4. After accepting a rotation, the device adds the new key and marks the old key as expired.

## Configuration

```toml
[attestation]
enabled = true
manifest_path = "/etc/wardex/manifest.json"
require_at_boot = true
periodic_check_minutes = 30
trust_store_path = "/etc/wardex/trust_store.json"
```

## Implementation phases

1. **v0.1** вҖ” Define `BuildManifest`, `TrustStore`, and `ArtifactEntry` structs in `src/attestation.rs`.
2. **v0.2** вҖ” Implement binary hash verification at boot (Linux `/proc/self/exe`).
3. **v0.3** вҖ” Implement artifact hash verification for baseline and config files.
4. **v0.4** вҖ” Build the `wardex-attest` CLI tool for manifest generation.
5. **v0.5** вҖ” Integrate into CI pipeline with Ed25519 signing.
6. **v0.6** вҖ” Add key rotation protocol.
7. **v0.7** вҖ” Add periodic runtime re-verification.
