Edge Security Runtime · Rust

SentinelEdge

Adaptive anomaly detection, energy-aware mitigation, and tamper-evident forensics for resource-constrained devices — packaged as a single Rust binary with no cloud dependency.

SentinelEdge runs the full detect–respond–audit loop locally across eight telemetry dimensions. The current codebase includes pluggable action adapters, SHA-256 audit chains, proof-carrying updates, a checkable policy state machine, poisoning heuristics, rollback checkpoints, forensic evidence bundles, and structured SIEM reporting. The larger research agenda — zero-knowledge attestation, swarm coordination, and post-quantum log integrity — is documented separately and left out of the runtime until there is real implementation behind it.

Architecture

System Design

The runtime follows a ten-stage pipeline. Each stage maps to one or more Rust modules with narrow interfaces. New complexity is added only when there is code to support it.

1

Ingest

Parse typed telemetry samples from CSV or JSONL. Validate field ranges. Auto-detect format by extension. Maintain deterministic replay semantics so test scenarios are reproducible.

telemetry.rs
2

Detect

Maintain a rolling EWMA baseline across eight signal dimensions (CPU, memory, temperature, bandwidth, auth failures, integrity drift, process count, disk pressure). Compute a weighted anomaly score with human-readable explanations. Persist and restore baselines across sessions.

detector.rs + baseline.rs
3

Respond

Map the anomaly score to one of four threat levels (nominal, elevated, severe, critical) and a corresponding response action. Degrade gracefully when battery state is constrained.

policy.rs
4

Execute

Dispatch actions through composable adapter chains. Logging, throttle, quarantine, and isolate adapters fire in sequence based on threat level.

actions.rs
5

Checkpoint

Capture detector state snapshots on severe/critical events. A bounded ring buffer retains recent checkpoints for post-incident analysis and forensic exports.

checkpoint.rs + forensics.rs
6

Audit

Append every decision to a SHA-256-chained audit log with signed checkpoints at configurable intervals. Verify the entire chain end-to-end.

audit.rs
7

Report

Emit structured JSON reports for SIEM ingestion, JSONL alert streams, and human-readable console summaries.

report.rs + config.rs
8

Proof

Bind every baseline state change to a SHA-256 proof linking prior state, transform, and post state. Accumulate and batch-verify proofs.

proof.rs
9

State Machine

Record and validate all threat-level transitions against formally defined legal rules. Export transition traces for future TLA+/Alloy verification.

state_machine.rs
10

Replay & Poisoning

Retain recent telemetry in a bounded ring buffer for windowed statistics. Four poisoning heuristics scan for data manipulation attempts.

replay.rs + poisoning.rs

Pipeline Internals

Telemetry is processed one sample at a time — this mirrors real-time constraints on edge hardware where batch processing is not always available. Below is a closer look at what each stage does and what it deliberately leaves out.

Implementation

Current Status

The project has completed all eight phases (0–7) of the engineering backlog. It provides a working edge security runtime with configurable detection, pluggable response actions, cryptographic audit trails, proof-carrying update metadata, a checkable policy state machine, poisoning heuristics, replay buffering, benchmark tooling, a live browser admin console backed by an authenticated HTTP API, and browser-based inspection of exported status/report artifacts. Phase 7 formalised research questions for all 15 expanded tracks (R26–R40) and produced design documents for adversarial robustness testing, temporal-logic monitoring, digital-twin simulation, and formal policy composition. Three categories below — no ambiguity about what is real versus what is planned.

        Setup

        Getting Started

        Requirements: a stable Rust toolchain (1.85+). No external services, no database, no cloud account.

        Build

        cargo build

        Run the demo trace

        cargo run -- demo

        Runs a short built-in telemetry sequence. Audit output goes to var/demo.audit.log.

        Analyze a scenario

        cargo run -- analyze examples/credential_storm.csv

        Full ingest → detect → respond → audit pipeline. Supports CSV and JSONL formats.

        Run tests

        cargo test

        52 automated tests covering all 17 modules.

        SIEM report

        cargo run -- report examples/credential_storm.csv

        Generates a structured JSON report for SIEM integration.

        Generate config

        cargo run -- init-config

        Writes a default TOML configuration template.

        Implementation snapshot

        cargo run -- status

        Browser console snapshot

        cargo run -- status-json site/data/status.json

        Exports the structured status file consumed by admin.html.

        Live admin console

        cargo run -- serve

        Starts the HTTP server on port 8080. Open localhost:8080/admin.html for the live control plane.

        Telemetry Formats

        The telemetry parser supports CSV (8-column legacy or 10-column extended) and JSONL. Format is auto-detected by file extension. CSV header:

        timestamp_ms,cpu_load_pct,memory_load_pct,temperature_c,network_kbps,auth_failures,battery_pct,integrity_drift[,process_count,disk_pressure_pct]
