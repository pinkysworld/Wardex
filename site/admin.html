Help & Documentation

Comprehensive documentation for the Wardex XDR platform — what is monitored, how detection works, and how to use every feature.

Getting Started
Quick Start Guide ▶

Installation

  1. Build the project: cargo build --release
  2. Start the server: ./target/release/wardex serve
  3. Copy the API token printed in the terminal
  4. Open the admin console at http://localhost:8080/admin.html
  5. Paste the token and click Connect

Configuration

Settings are stored in var/wardex.toml. Edit via the Settings panel or the TOML file directly. The server picks up changes on config reload.

Your First Host

The server automatically monitors the local machine. Once connected, the Dashboard shows real-time CPU, memory, network, and other telemetry. Alerts appear when anomaly scores exceed the configured thresholds.

Demo Mode ▶

What Is Demo Mode?

Demo mode simulates a realistic XDR detection scenario entirely in the browser. It generates synthetic telemetry showing a normal baseline followed by an escalating attack.

How To Use

  1. Toggle the “Demo Mode” switch in the top bar
  2. Navigate to the Dashboard to see synthetic gauges and sparklines
  3. Watch the Live Monitor for simulated alerts
  4. Toggle off to return to real data

What Gets Simulated

  • CPU ramp from 15% to 95% (simulated cryptominer)
  • Memory climbing from 40% to 88%
  • Network spike to 50,000 packets (data exfiltration)
  • Authentication failure burst (credential stuffing)
  • Alerts escalating from Elevated through Severe to Critical

Note: Demo mode runs entirely client-side. No data is sent to the server and no state is modified.

Platform Support ▶

Supported Platforms

  • macOS — Full support. Telemetry via sysctl, vm_stat, netstat, IOKit, pmset, log show.
  • Linux — Full support. Telemetry via /proc filesystem, /sys/class, journalctl.
  • Windows — Partial support. Telemetry via performance counters, WMI, Event Log.

Service Installation

  • macOS — LaunchDaemon plist in /Library/LaunchDaemons/
  • Linux — systemd unit file in /etc/systemd/system/
  • Windows — Windows Service via sc.exe
Detection & Architecture
What Is Monitored ▶

The Wardex monitor collects the following telemetry from each host in real-time:

CPU Usage

Percentage of CPU time in use. On macOS via sysctl kern.cp_time; on Linux via /proc/stat. Spikes may indicate cryptomining, brute-force attacks, or runaway processes.

Memory Pressure

Percentage of physical RAM in use. On macOS via vm_stat; on Linux via /proc/meminfo. Sudden increases can signal memory exhaustion attacks or malware injection.

Network Activity

Total packets transmitted across all interfaces. Anomalous spikes may indicate data exfiltration, C2 beacons, or DDoS activity.

Disk Pressure

Percentage of disk space used on the root filesystem. Rapid consumption can indicate ransomware encryption or log bombing.

Temperature

CPU or SoC temperature in degrees Celsius. Elevated temperatures may indicate cryptojacking or hardware tampering.

Authentication Failures

Count of failed login attempts. Spikes indicate brute-force or credential-stuffing attacks.

Battery Level

Current battery percentage (portable/IoT devices). Low battery can be weaponized to disable monitoring.

Process Count

Number of running processes. Sudden increases may indicate fork bombs, lateral movement, or persistence mechanisms.

File Integrity Monitoring (FIM)

SHA-256 hashes of critical system files. Any hash change generates an immediate alert — detects rootkits and supply-chain tampering.

Drift Detection

Kolmogorov-Smirnov test tracks how telemetry distributions change over time, detecting slow stealthy attacks.

How Detection Works ▶

EWMA Anomaly Detection

The core engine uses an Exponentially Weighted Moving Average (EWMA) to establish dynamic baselines for each telemetry dimension. Each sample produces a single numeric anomaly score as the sum of normalized deviations.

Multi-Axis Scoring

10 independent axes are evaluated: CPU, memory, network, disk, temperature, auth failures, battery, process count, file integrity, and custom metrics. Each axis maintains its own mean and variance.

Detection Modes

  • Normal — Continuously learns and updates baselines.
  • Frozen — Uses the current baseline without updating.
  • Decay — Gradually forgets old baselines at a configurable rate.

Threat Levels

  • Nominal — Score below elevated threshold (default: 1.4)
  • Elevated — Score between 1.4 and 3.0
  • Severe — Score between 3.0 and 5.2
  • Critical — Score above 5.2

Velocity Detection

Tracks the rate of change and acceleration of each telemetry axis over a sliding window. When velocity exceeds the historical mean by >σ standard deviations, it flags a velocity spike — detecting rapid ramp-ups even when absolute values remain below static thresholds.

Entropy Analysis

Computes Shannon entropy (H = −∑ p·log₂p) over a sliding window. Low entropy indicates suspiciously constant traffic (cryptominers, DDoS floods). High entropy on auth-failures indicates randomised credential stuffing.

Compound Threat Detection

Identifies coordinated multi-axis attacks. When ≥40% of axes are simultaneously elevated, the attack is classified as a compound threat and receives a score multiplier.

Policy & Correlation

The policy engine maps scores to enforcement actions. Policies can be composed and executed in a sandboxed WASM VM. Cross-agent correlation detects coordinated attacks spanning multiple endpoints.

XDR Agent Architecture ▶

Server + Agent Model

The server monitors the local system automatically. Remote hosts run lightweight agents that report telemetry to the central server.

Agent Enrollment

Agents enroll using one-time tokens generated from Fleet & Agents → Create Enrollment Token. Each token has configurable max-uses. Agents authenticate on each heartbeat.

Heartbeat & Telemetry

Agents send heartbeats at configurable intervals (default: 5s). Each heartbeat includes a full telemetry sample evaluated through the anomaly detector.

SIEM Integration

Events can be forwarded to external SIEM platforms via webhook, syslog, or CEF format.

Policy Distribution

Security policies are versioned and distributed to agents via the policy store on each heartbeat check-in.

Reference
CLI Commands ▶

Available commands when running wardex:

  • start — Server + live monitor (default, no args needed)
  • serve [port] [site_dir] — HTTP API server with admin console
  • server [port] — XDR central server mode
  • monitor [flags] — CLI-only host monitoring
  • agent --server <url> --token <t> — XDR agent mode
  • analyze <file> — Analyze a CSV/JSONL telemetry file
  • report <file> — Generate a JSON detection report
  • bench <benign> <attack> [thresh] — Benchmark detectors
  • demo [audit] — Run demo telemetry analysis
  • status — Print runtime status manifest
  • status-json [path] — Export status as JSON
  • init-config [path] — Write default config file
  • attest <binary> — Generate build manifest
  • harness — Run adversarial harness
  • export-model <tla|alloy> — Export formal model
  • help — Show usage
